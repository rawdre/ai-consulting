## 5. Delegation Rules
When one bot delegates to another:
1. define the task clearly
2. define the expected output
3. define the destination folder/file
4. do not create parallel folder systems without approval
5. merge results back into the canonical structure

### No parallel chaos
Bots must **not**:
- invent competing folder structures
- publish from random repos/branches
- duplicate files across multiple roots without reason
- silently change canonical ownership of projects

## 7. GitHub Signature Rules
To keep authorship clear, use visible signatures in 2 layers.

### A. In-document/page signature
At the bottom of docs/pages/notes when relevant:
- `Organized by Rawbot 🔥`
- `Crafted by André + Acerbot`
- `Structured with LenoRawbot`

Only sign where there was real contribution.
Do not fake authorship.

### B. Git commit style
Recommended commit prefixes:
- `RAWBOT:` for organization, coordination, workspace, governance, deploy-flow
- `ACERBOT:` for monetization, packaging, offers, commercial systems
- `LENO:` for diagnostics, structure, continuity, process cleanup

Examples:
- `RAWBOT: organize workspace and define canonical publish flow`
- `ACERBOT: package ai-consulting offer into retainer model`
- `LENO: clean operational docs and improve system continuity`

### C. Commit messages should reflect responsibility
If a bot touched a commit meaningfully, the commit should make that visible.
